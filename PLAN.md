# koe-survey（コエ）— 汎用エンゲージメント／NPS測定Webアプリ PLAN

**作成日**: 2026-07-07
**目的**: クライアント横断で使い回せる「アンケート配信＋匿名回答＋自動集計ダッシュボード」を1本作る。
**名前の由来**: 西尾歯科NPS設計の大原則「物品でつるのではなく、声が活きる仕組みで動機を作る」→「声（コエ）」。

---

## 1. 誰の・何の課題を解くか

| クライアント | 使い方 | 時期 |
|---|---|---|
| 西尾歯科 | 患者NPS15問（ユニット別QR×3）＋スタッフ・エンゲージメント10問 | 7/13訪問で提案 |
| コスモス | 職員エンゲージメント10問＋ハラスメント設問（7月実施目標） |7月 |
| 北野田診療所 | 体験者win/lossアンケート・ストレスチェック（将来） | 現場温度を見て |
| 同仁会 | 「対話の時間」3ヶ月ごとアンケート（将来） | 秋以降 |

**共通の型**: 「QRコード→スマホで匿名回答（3〜5分）→自動集計→ダッシュボードで経年比較」。
今は紙・Word・GASでバラバラ→1本の仕組みに集約する。

---

## 2. デザイン原則（変えない約束）

1. **設問はデータ、アプリはエンジン** — 設問セットはJSON定義。コードを触らずに新しいアンケートを追加できる（schema-driven。collectionsと同じ思想）。
2. **匿名ファースト** — 回答に個人特定情報を持たない。IP・氏名・メール不収集。回答日時は「日」まで丸める。属性（部署等）は「n<5なら表示しない」を集計側で強制。※例外：西尾歯科NPSのQ15診察券番号のように、設問として任意で聞くものはOK（設問定義で明示）。
3. **回答者の摩擦ゼロ** — ログイン不要・アプリインストール不要・スマホ1画面ずつ・3〜5分。QRを読んだら即回答画面。
4. **集計は自動、解釈は人間** — スコア計算（NPS・ERG軸別平均・経年推移）はアプリ。「どう打ち手につなげるか」は平井さんのコンサルの仕事として残す（アプリはコンサルの道具、代替ではない）。
5. **クライアントごとに閉じる** — surveyはトークン付きURLで隔離。管理画面はログイン必須。クライアントAから他クライアントのデータは見えない。
6. **運用コストゼロ円** — GitHub Pages（静的）＋Supabase無料枠。サーバー管理なし。

## 3. アーキテクチャ

```
[回答者スマホ]                       [平井さん／クライアント管理者]
     │ QRコード                            │ ログイン（Supabase Auth）
     ▼                                     ▼
GitHub Pages（静的SPA・素のHTML/JS）────────────────
  /#/s/<survey-slug>?ch=unit1   /#/admin
     │ INSERTのみ（anonキー＋RLS）    │ SELECT（認証済のみ・RLS）
     ▼                              ▼
Supabase（Postgres）
  surveys / responses
```

- **フロント**: 素のHTML/CSS/JS（ビルド不要・依存最小）。1リポジトリ＝1デプロイ。
- **バックエンド**: Supabase。回答は匿名キーで**INSERTのみ可**（RLSでSELECT/UPDATE/DELETE禁止）。集計読み出しは**認証ユーザーのみ**。
- **ホスティング**: GitHub Pages（`gh-pages`不使用、`main`の`/docs`か root。→ rootでシンプルに）。
- **QRコード**: 管理画面でクライアントサイド生成（qrcode.jsをリポジトリに同梱。CDN依存しない）。

### なぜGoogle Formでなくこれか
ユニット別URL・ERG軸の自動レーダー・NPS計算・経年比較・「クライアントに見せられる画面」が一体になったものは既製にない。かつ2027年〜のAIサポートサービスの実績第1号になる。

## 4. データ構造（人間が握る1%）

### surveys テーブル（設問セット＝JSON定義）
```sql
surveys (
  id          uuid PK default gen_random_uuid(),
  slug        text UNIQUE,          -- URL用: 'nishio-nps-2607'
  client      text,                 -- 'nishio' | 'cosmos' | ...
  title       text,
  status      text default 'draft', -- draft | open | closed
  definition  jsonb,                -- 設問定義（下記）
  created_at  timestamptz default now()
)
```

### definition JSONスキーマ（アプリの心臓部）
```jsonc
{
  "intro": "3分ほどで終わります。回答は匿名です。",
  "channels": ["unit1", "unit2", "unit3"],   // 配布経路（QR別）。無ければ省略
  "anonymous_note": "回答は個人を特定できない形で集計されます",
  "questions": [
    { "id": "q1", "type": "choice",  "text": "本日の来院目的", "required": true,
      "options": ["メンテナンス", "治療", "自費治療", "急患", "その他"] },
    { "id": "q2", "type": "nps",     "text": "西尾歯科をご家族やご友人に勧めたいと思いますか？", "required": true },
    { "id": "q3", "type": "text",    "text": "上記の点数の理由" },
    { "id": "q4", "type": "likert5", "text": "受付スタッフの挨拶・笑顔はいかがでしたか？",
      "low": "気になった", "high": "とても気持ち良かった",
      "axis": "R" },                          // axis: ERG軸タグ（E/R/G）。集計でレーダー化
    ...
  ],
  "scoring": {
    "nps_question": "q2",            // NPS計算対象
    "headline": "q10",               // ヘッドライン設問（例: 働き続けたい）
    "erg_axes": true                 // axisタグの軸別平均を出す
  }
}
```

**設問type一覧（v1）**: `choice`（単一選択）/ `nps`（0〜10）/ `likert5`（5段階・低高ラベル）/ `text`（自由記述）。
これで手持ちのNPS15問・ERG10問・体験者インタビュー票がすべて表現できることを確認済み。

### responses テーブル
```sql
responses (
  id          uuid PK default gen_random_uuid(),
  survey_id   uuid REFERENCES surveys,
  channel     text,                 -- QRのch=パラメータ
  answers     jsonb,                -- { "q1": "メンテナンス", "q2": 9, ... }
  answered_on date default current_date   -- 「日」まで。時刻は持たない（匿名性）
)
```

### RLS（守りの要）
- `responses`: anonロールは **INSERTのみ**（かつ対象surveyが`status='open'`のときのみ）。SELECT不可。
- `surveys`: anonロールは `status='open'` の行の `definition` 等のみSELECT可。
- 認証済みユーザー（平井さん）: 全権。クライアント管理者を将来足す場合は `client` でRLS絞り込み（v2）。

## 5. 画面構成

### 回答画面 `/#/s/<slug>?ch=<channel>`
- イントロ（所要時間・匿名の明記）→ 設問を1問ずつ or 1ページ縦スクロール（**v1は縦スクロール1ページ**。歯科の待合で途切れても戻れる方が大事）
- NPSは0〜10の横並びボタン、likert5は5ボタン＋両端ラベル
- 送信→サンクス画面（「声は改善に活かします」）
- 多重送信防止: localStorageに送信済みフラグ（完全防止はしない＝匿名優先）

### 管理画面 `/#/admin`（Supabase Authログイン）
1. **サーベイ一覧** — クライアント別、status、回答数
2. **サーベイ詳細（ダッシュボード）**
   - NPSスコア（推奨者%−批判者%）＋回答数＋月次推移（折れ線）
   - ERGレーダー（E/R/G軸別平均）＋設問別平均（横棒・低い順）
   - ヘッドライン設問の推移
   - チャネル別内訳（unit1/2/3）
   - 自由記述一覧（新しい順）
   - **CSVエクスポート**（詳細分析・報告書用）
   - **n<5の属性クロスは非表示**の但し書き
3. **QRコード表示** — チャネル別QRを表示・印刷用（A6カード風レイアウト）
4. サーベイ作成・編集はv1では**JSONを直接貼る**簡易エディタ（設問デザインはClaude Codeで作ってコピペする運用。GUIエディタはv2）

### デザイントーン
西尾歯科・コスモスの既存成果物と同系（白ベース・青アクセント・游ゴシック系）。クライアントに見せて恥ずかしくないこと。ダークモード不要（v1）。

## 7. セキュリティ・プライバシー覚え書き

- Supabaseの**anonキーはフロントに露出してよい**（公開前提のキー。守りはRLSが担う）。service_roleキーは**絶対にリポジトリに入れない**。
- 回答データに個人情報を原則持たないので、漏えい時の被害面を最小化。
- 診察券番号のような準識別子を設問に含めるsurveyは、CSVエクスポート時に取り扱い注意の表示。
- リポジトリは**public**（GitHub Pages無料枠のため）。データはSupabase側なのでコード公開に支障なし。クライアント名はslugに出るため、気になるなら`n-dental`等の仮名slugにする。
