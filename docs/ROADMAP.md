## 2. 戦略の柱（v2での練り直し）

v1は「西尾歯科NPSを7/13に見せる」ための最短距離だった。v2からは以下の4本柱で考える。

- **柱A：汎用化** — どのクライアントにも「設問JSONを1つ作るだけ」で展開できる状態を保つ。コードにクライアント固有の分岐を作らない（schema-driven原則の徹底）。
- **柱B：コンサル価値の増幅** — 集計の自動化はアプリ、解釈と打ち手は平井さん。アプリの伸ばす方向は「解釈の材料を増やす」こと（1枚レポート・自由記述の整理・経年比較）。競合SaaSとの差別化は「ツール＋コンサル一体」のモデル自体にある。
- **柱C：回答率の最大化** — 現場（待合室・職場）での回答のしやすさに投資する。リサーチの知見では「1画面1問」「キオスク型の常設端末」が完了率を大きく上げる。
- **柱D：運用の堅牢さ** — 無料プラン運用の弱点（7日停止・バックアップなし・Actions60日停止）を仕組みでカバーし続ける。

## 3. 汎用化仕様（他企業への展開）

### いまの設計で既に汎用的なもの（変更不要）
- 設問セット＝JSON定義（choice / nps / likert5 / text の4タイプで手持ちの全票が表現できる）
- `client` フィールドによるクライアント識別（仮名運用）
- チャネル別QR（`?ch=`）による配布経路の分離
- ERG軸タグ・NPS・ヘッドライン設問のスコアリング設定

### 追加すべき汎用化（提案。実装は各WPで）
1. **設問テンプレライブラリ**（WP-4）— `surveys/templates/` に業種別テンプレJSONを蓄積：歯科NPS／医療職員ERG／介護職員／一般企業eNPS／ハラスメント設問／体験者win-loss。新規クライアントは「テンプレを選ぶ→文言を微調整→slug発行」の3ステップで開始できる。**コード変更なしのデータ追加**なので安全に増やせる。
2. **ブランディング設定**（WP-8・要承認）— `definition.branding = { display_name, accent_color }` を追加し、回答画面の見た目をクライアントごとに変えられるようにする。⚠️ definition JSONスキーマの変更にあたるため、**実装前に平井さんの承認を得ること**（ガードレール：データ構造変更は提案どまり）。
3. **文言の変数化はしない**（現行方針維持）— 「当院」「法人」などはクライアント別にJSONをコピーして調整する。変数化はテンプレの可読性を下げるため見送り。
4. **多言語対応は v3 以降** — 需要が出てから（外国人患者の多い医院等）。

## 4. リサーチ結果：機能候補と採用判断

2026-07-07に競合・類似サービスを調査した結果。参考: [国内エンゲージメントサーベイ比較](https://boxil.jp/mag/a9009/)・[患者体験ツール](https://www.zonkafeedback.com/blog/patient-experience-tools)・[キオスク型サーベイ](https://www.zonkafeedback.com/kiosk-surveys)・[Delighted機能一覧](https://delighted.com/features)

| 機能 | 参考サービス | 判断 | 理由 |
|---|---|---|---|
| キオスクモード（回答後自動リセット・待合室タブレット常設） | Zonka / SurveyLegend | **labs試作済 → WP-5** | 紙のアンケートを置き換える導線。歯科・診療所と相性が良い |
| 1枚レポート（A4印刷・月次サマリー） | Delighted | **labs試作済 → WP-3** | 平井さんの報告書作成を数時間→数分に。コンサル価値に直結 |
| 自由記述インサイト（頻出語・テーマ抽出） | Delighted Smart Trends | **labs試作済 → WP-6** | 自由記述が溜まるほど読む負担が増える問題への先手。**ブラウザ内処理のみ**（外部送信なし＝匿名性維持） |
| パルスサーベイ（週1問・1画面1問） | Wevox / Surveyapp | **labs試作済 → WP-9（要承認）** | 職員エンゲージメントの「測りっぱなし防止」。回答率向上の知見（1画面1問）も適用 |
| QRコード生成（管理画面内） | 各社標準機能 | **WP-2**（PLAN既載） | 現在は外部ツール頼み。qrcode.jsを同梱しCDN非依存で |
| 回答率トラッカー（配布数手入力→回答率） | 各社標準機能 | **WP-7** | 「配って終わり」を防ぐ小粒機能 |
| 業界ベンチマーク | Delighted / モチベーションクラウド | **保留** | 自社データが溜まってから「クライアント横断の匿名ベンチマーク」として検討。n確保と匿名性の設計が必要 |
| 期待度×満足度の2軸診断 | モチベーションクラウド | **保留** | 設問数が倍増し「3〜5分で終わる」原則と衝突。コンサル側の分析手法として先に試す |
| SMS/メール配信・リマインド | 各社標準機能 | **不採用** | 個人の連絡先を持つこと自体が匿名ファースト原則と衝突。QR掲示＋紙POP運用で代替する |
| AI要約（LLMによる自由記述の要約） | Wevox AIサポート | **保留** | 匿名でも回答データを外部APIへ送る是非を先に平井さんと決める。まずWP-6（ブラウザ内頻出語）で価値検証 |
| ストレスチェック様式 | ラフール等 | **v3**（PLAN既載） | 法定様式対応は要件が重い。需要（北野田）の温度を見て |

## 5. 作業パッケージ（WP）— Opus実行用

### 実行ルール（全セッション共通）
1. **1セッション＝1WP**。複数WPを1つのPRに混ぜない。
2. 着手前に **PLAN.md → CLAUDE.md → 本書** の順で読む。
3. 成果は**PR1本**にまとめ、**動作スクリーンショット付き**で出す（`python3 -m http.server 8000` → Playwright/Chromiumはこの環境に同梱）。
4. ガードレール（匿名性・RLS・実名禁止・デモモード維持・チャート配色）に触れる変更は**実装せず提案として書く**。
5. 「要承認」マークのWPは、着手前に平井さんへ仕様の承認を取る。

### WP一覧（優先順）

**WP-0：本番接続の再開** — 保留中
- 内容: PR #3 に `sb_publishable_` キーの差し替えコミットを追加→マージ→keep-alive成功（HTTP 200）を確認
- 触るファイル: `js/config.js` のみ
- 完了条件: Actionsの「Supabase keep-alive」が緑／本番 `#/admin` にログインできる
- 依存: 平井さんからのキー提供

**WP-1：本番サーベイ登録（運用作業・コード変更なし）**
- 内容: nd（西尾歯科の仮名）向けNPS15問を本番Supabaseに登録し、チャネル別URL（unit1〜3）を発行
- 完了条件: 本番URLでスマホから回答→管理画面に反映される
- 依存: WP-0

**WP-2：QRコード生成**
- 内容: qrcode.js系ライブラリをリポジトリに同梱（CDN不使用）し、管理画面のチャネル別URL欄にQR画像＋印刷用レイアウト（A6カード風）を追加（PLAN §5-3）
- 触るファイル: `js/admin.js`, `css/style.css`, 同梱ライブラリ1ファイル
- 完了条件: 管理画面でQRが表示・印刷でき、スマホで読み取って回答画面が開く

**WP-3：1枚レポート印刷ビュー**
- 内容: `labs/report.html` を本実装。管理画面ダッシュボードに「印刷レポート」ボタン→A4・1枚の月次サマリー（NPS・前月差・推移・設問別平均・自由記述抜粋・n<5注記）
- 触るファイル: `js/admin.js`, `js/charts.js`（流用）, `css/style.css`（@media print）
- 完了条件: ブラウザの印刷でA4・1枚に収まる／デモ30件で崩れない

**WP-4：設問テンプレライブラリ**
- 内容: `surveys/templates/` に業種テンプレJSON（§3参照）を追加し、新規サーベイ作成画面にテンプレ選択→JSONエディタへ流し込み
- 触るファイル: `surveys/templates/*.json`, `js/admin.js`
- 完了条件: テンプレ選択→slugとclientだけ変えて登録→回答できる

**WP-5：キオスクモード**
- 内容: `labs/kiosk.html` の知見を本体へ。回答URL に `&kiosk=1` を付けると (1)送信後サンクス画面を数秒表示して自動で先頭に戻る (2)一定時間無操作でもリセット (3)多重送信フラグ（localStorage）をキオスクでは使わない
- 触るファイル: `js/survey.js`, `css/style.css`
- 完了条件: タブレットを想定した連続回答が手を触れずに回る

**WP-6：自由記述インサイト**
- 内容: `labs/text-insights.html` の本実装。ダッシュボードの自由記述欄に頻出キーワードランキング（クリックで該当コメント絞り込み）。処理は**全てブラウザ内**（外部送信なし）
- 触るファイル: `js/admin.js`, `css/style.css`
- 完了条件: デモ30件で頻出語が表示され、クリックで絞り込める

**WP-7：回答率トラッカー**
- 内容: サーベイごとに「配布数」を手入力（surveysテーブルのdefinition内 or 管理画面のローカル設定）→ダッシュボードに回答率タイル表示
- 完了条件: 配布数を入れると回答率が出る。未入力なら非表示

**WP-8：ブランディング設定 ⚠️要承認**
- 内容: `definition.branding = { display_name, accent_color }`。回答画面のタイトル・アクセント色に反映
- 注意: definition スキーマ変更のため、**着手前に平井さんの承認必須**。チャートの検証済み配色は変えない（管理画面側は現行のまま）

**WP-9：パルス型回答UI ⚠️要承認**
- 内容: `definition.mode: "pulse"` のサーベイは1画面1問のカード型UIで表示（labs/pulse.html参照）。既存の縦スクロール型は不変
- 注意: 同じくスキーマ変更のため要承認

## 6. labs/（独立プロトタイプ置き場）

- `labs/` は**本体と完全に独立した**単一HTMLのプロトタイプ置き場。本体のJS/CSSを一切importせず、デモデータ内蔵。`index.html` や `js/` からリンク・参照しない。
- 目的: 機能のアイデアを「動くもの」でクライアント・平井さんに見せ、採否を判断する。
- **昇格基準**: 平井さんが実クライアントで使うと判断したら、対応するWPとして本体に実装し直す（labsのコードはコピペせず、本体の作法＝charts.js・style.css・api.js に合わせて書き直す）。
- 不要になったプロトタイプは削除してよい。
